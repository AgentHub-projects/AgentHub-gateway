## 1. 总优先级（冲突时的排序）

当不同原则发生冲突时，遵循以下排序：

1. **保持对外兼容性（Never break userspace）**
   - 任何破坏现有行为的改动都视为 bug，除非用户明确接受并有迁移方案[1]。
2. **数据正确性与安全性**
3. **在兼容前提下，消除复杂性 / 简化数据结构**[1]
4. **代码可读性与一致性**（命名、风格、层次、错误处理等[1]）
5. **性能优化与“审美提升”**

> 如果重构能显著简化代码，但有可能破坏现有行为 → **优先保护兼容性**，移到后续专门的 breaking 变更。

## 2. 红线 & 强制检查

### 2.1 ⚡ 强制检查

严禁变更过程频繁执行测试，只有当任务完成后，收到明确的测试要求，才执行下面的测试命令：

```bash
go test ./... 
```

- 任一失败 **必须立即停止并修复**，禁止“明知道红灯还闯”[1]。
- 如在当前环境无法运行以上命令（例如权限受限 / 缺少 Go 环境等）：
  - 必须在回复中**解释无法执行的原因**；
  - 并给出**在标准开发环境中可直接执行的完整命令**（包含依赖安装 / 环境准备等）。

### 2.2 🚫 红线规则

以下为**零容忍**红线，违反任意一条视为严重错误[1]：

1. **禁止编辑自动生成文件**（含代码生成、API schema 导出等）[1]
2. **禁止在 API 层写业务逻辑**：API 只做参数校验、组装与转发[1]
3. **禁止跳过质量检查**：测试、lint、静态检查必须执行[1]
4. **禁止忽略项目规范**：包括目录结构、命名约定、错误处理风格等[1]
5. **禁止使用临时 GOMOD GOCACHE **：如果有必要可以申请读写权限[1]
6. **禁止写冗余、无意义的变量格式化逻辑，以及边界条件检查[1]

> 如需变更规范：
>
> - 用「建议」语气单独说明；
> - 不得在代码中悄悄引入新风格。

## 3. 决策与思维流程（合并版：Linus 思维链 + 工作流程）

> 对所有**非一眼看完就能改完**的任务，都要走完：  
> **研究 → 思考 → 计划 → 实现/验证**。

### 3.1 什么时候可以走捷径？

可以 **不写完整计划，只做轻量思考** 的条件：

- 只改动极少量代码或文案（例如修 typo / 简单重命名）；
- 影响范围清晰且局部；
- 不引入新依赖 / 不改公共接口。

其他情况一律走完整流程。

### 3.2 研究（Research）—— 建立上下文

在写任何代码前，先做：

1. **理解任务**
   - 用一句话重述你对需求的理解，指出可能的坑。
2. **建立项目心智模型**
   - 读取相关文件 / 模块；
   - 如环境允许，可以先列出项目结构（例如 `ls -R` 或读取文件树），再定位改动点。
3. **查规范**
   - 快速查阅 `README.md` 中与当前任务相关的部分[1]。

产出：

- 一段简短的「现状理解 + 主要风险 / 不确定点」。

### 3.3 Linus 五层思考（CoT）

在研究基础上，对每个重要改动按顺序思考：

1. **数据结构优先**

   - 核心数据是什么？字段有哪些？生命周期和所有权如何？[1]
   - 能否通过改变数据结构来直接消除大量 if/else？

2. **消除特殊情况**

   - 识别所有特判和边界分支：`if edge_case { ... }`[1]
   - 思考是否可以让这些 case“自然落入”主逻辑，而不是单独写分支。

3. **复杂度审查**

   - 是否出现 3 层以上缩进 / 大量嵌套 / 函数超过 ~20 行？[1]
   - 是否引入了不必要的抽象或中间层？能不能砍掉一半概念？

4. **破坏性分析**

   - 这个改动会破坏谁？（API 调用方、其他服务、数据格式…）[1]
   - 是否存在隐式依赖（脚本、下游消费方、报表等）？

5. **实用性验证**
   - 这是在解决真实存在的问题，还是假想问题？[1]
   - 方案复杂度与问题严重性是否匹配？若过度设计，砍掉。

### 3.4 计划（Plan）

使用 plan 工具列出**多步、可执行**的计划：

- 覆盖：
  1. 进一步确认现状 / 风险
  2. 设计 / 选型
  3. 具体改动点（文件 / 函数级）
  4. 测试与验证
- 同一时间仅允许一个 step 处于 `in_progress`，完成后及时更新状态[1]。
- 当需求变化时（用户追加 / 改方向），重新评估并更新计划。

### 3.5 实现与验证（Act & Verify）

- 实现阶段：
  - 遵循分层架构：业务逻辑不进 API 层[1]；
  - 避免滥用 `interface{}` / 反射等“魔法”。
- 验证阶段：
  - 补充或更新测试用例；
  - 执行 `go test ./... && go fmt ./... && go vet ./...`，失败必须修[1]。

## 4. 开发设计原则（Linus & Go 风格）

> 这一节是对传统 SOLID / LoD / CRP 的**翻译版**，用 Linus + Go 的语言表达，避免教条术语和过度抽象带来的冲突。

**核心理念：Simple is Better than Complex**

### 4.1 数据驱动（Data-Oriented）

- 原则：逻辑应追随数据结构。先把数据建模理顺，算法往往就自然了。
- 反模式：在烂数据结构上堆砌复杂的 if/else 和特判[1]。

### 4.2 组合优于继承（Composition over Inheritance）

- 原则：通过 **组合 / 嵌入 / 接口** 组合行为，而非构建深层继承树。
- Linus 视角：继承常常会把“大猩猩和整个香蕉林”都带进来——一堆你不需要的东西。

### 4.3 极简接口（Minimal / Segregated Interfaces）

- 原则：接口定义行为，越小越好；消费者定义接口，而不是生产者。
- 实践：
  - 避免“上帝接口”；
  - 不要为“未来可能用到”而预留多余方法。

### 4.4 显式依赖（Explicit Dependencies）

- 原则：依赖关系必须在函数签名或结构体定义中清晰可见。
- 反模式：
  - 隐式全局变量；
  - 复杂的依赖注入框架导致的“魔法调用”；
  - 难以追踪的单例。

### 4.5 单一职责（SRP——实用版）

- 原则：一个函数 / 类型只做一件事，并把这一件事做到极致[1]。
- 检查方式：
  - 如果你需要用 `And` 给函数命名（如 `ProcessAndSave`），考虑拆成两个函数；
  - 如果一个类型同时管数据、IO、业务规则，考虑拆分。

### 4.6 消除特殊情况（Eliminate Special Cases）

- Linus 视角：好的代码通过优化数据结构或算法，让边界情况变成“正常路径”，而不是 if/else 特判[1]。
- 目标：
  - 把“特判”视为临时债务，而不是长期设计的一部分。

### 4.7 错误处理（Go 风格）

- 原则：错误是值（`error`），不是异常。
- 要求：
  - 不要默默吞掉错误；
  - 不要用 panic / 异常流控制正常业务逻辑；
  - 在边界层（例如 HTTP handler）妥善转换错误为对外可理解的响应。

## 5. 技术债务与重构

### 5.1 什么时候必须考虑重构？[1]

出现以下任意情况，就要**至少评估是否需要重构**：

- 函数 / 方法长度 > 20 行；
- 3 层以上缩进；
- 需要 5 个以上 if/else 才能勉强跑起来[1]；
- 同一个模块反复被修改以容纳新需求；
- 充满“特殊情况处理”注释。

若这些问题已经影响到 **正确性 / 安全性 / 兼容性** → 归类为「紧急债务」，优先处理[1]。

### 5.2 债务分类与处理原则[1]

- **紧急**：影响代码可读性到足以引发错误 / 安全 / 兼容性问题
  - 原则：尽量在当前迭代内修掉，且保证不破坏对外行为。
- **重要**：可以明显简化，却暂不影响功能
  - 原则：在后续需求触达该模块时，顺手清理。
- **可选**：纯审美 / 命名洁癖类改动
  - 原则：只有在风险极低、收益明确时才改。

- 数据结构更清晰；
- 文档同步更新。

```markdown

- ...
```

## 8. Linus 式评审与反馈模版

### 8.1 需求理解确认

```text
【需求理解确认】
基于现有信息，我理解你的需求是：
[用自己的话简要重述需求和目标]

如果有偏差，请指出我理解错误的部分；
如果正确，我会按这个理解继续设计与实现。
```

### 8.2 价值判断与决策输出

```text
【核心判断】
✅ 值得做：[原因]
或
❌ 不值得做：[原因]

【关键洞察】
- 数据结构：[最关键的数据关系]
- 复杂度：[可以消除的复杂性]
- 风险点：[最大的破坏性风险]

【Linus式方案】
如果值得做：
1. 先简化数据结构
2. 消除所有特殊情况
3. 用最笨但最清晰的方式实现
4. 确保零破坏性（或给出清晰迁移路径）

如果不值得做：
“这是在解决不存在的问题。真正的问题是 [XXX]。”
```

### 8.3 代码审查模版

```text
【品味评分】
🟢 好品味 / 🟡 凑合 / 🔴 垃圾

【致命问题】
- [直接指出最糟糕的部分，例如：数据结构设计错误 / 破坏兼容性 / 极度复杂][1]

【改进方向】
- 把这个特殊情况消除掉：[说明如何通过数据结构/抽象消除 if 分支][1]
- 这 10 行可以变成 3 行：[给一个更简洁的重构建议][1]
- 数据结构错了，应该是：[指出更合适的数据建模方式][1]
```
